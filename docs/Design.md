# 랜딩페이지 디자인 가이드

> 대상: 건설 현장 기성 청구 자동화 SaaS 수요 검증 랜딩페이지
> 기준 문서: `plans/landing_page_demand_validation_plan.md`
> 작성일: 2026-05-12
> 디자인 방향: Apple식 명료함과 절제된 프리미엄감, B2B 현장 업무 신뢰감

---

## 1. 디자인 목표

랜딩페이지는 방문자가 예쁜 서비스를 감상하게 만드는 화면이 아니라,
원도급 공무, 현장소장, 감리단이 다음 문장을 빠르게 이해하고 파일럿 신청
여부를 판단하게 만드는 화면이다.

> 현장 기록이 곧 기성 승인 자료가 된다.

따라서 디자인의 1순위는 장식이 아니라 다음 세 가지다.

1. 첫 화면에서 제품의 정체와 가치를 5초 안에 이해시킨다.
2. "승인 가능 시공 데이터"라는 새로운 포지셔닝을 신뢰 가능하게 보인다.
3. 파일럿 신청 CTA와 폼 제출 흐름을 방해하는 요소를 줄인다.

## 2. 최신 디자인 리서치 반영

### 2-1. Apple Human Interface Guidelines에서 가져올 원칙

Apple HIG의 핵심은 단순히 흰 배경, 큰 글자, 둥근 모서리가 아니다. 이
랜딩페이지에는 다음 원칙만 반영한다.

- **Hierarchy**: 가장 중요한 메시지, 제품 시각화, CTA가 명확한 순서로 보인다.
- **Harmony**: 여백, 타이포그래피, 버튼, 섹션 간격이 같은 리듬을 갖는다.
- **Consistency**: 같은 행동은 같은 모양으로 보이고, 같은 수준의 정보는 같은
  밀도로 배치한다.
- **Adaptability**: 모바일, 태블릿, 데스크톱에서 메시지와 CTA가 잘리지 않는다.
- **Progressive disclosure**: 모든 설명을 한 번에 늘어놓지 않고, 핵심 요약 후
  필요한 근거를 단계적으로 보여준다.

참고: Apple Human Interface Guidelines
https://developer.apple.com/design/human-interface-guidelines

### 2-2. 2026 커뮤니티/업계 트렌드에서 반영할 점

최근 Webflow 커뮤니티와 랜딩페이지 디자인 글의 공통 신호는 다음이다.

- AI로 만든 듯한 과장된 그래픽보다 **의도적으로 편집된 짧은 카피**가 더 강한
  신뢰 신호가 된다.
- 미니멀 디자인은 여전히 유효하지만, B2B 고관여 제품은 **정보 밀도와 스캔
  가능성**이 함께 필요하다.
- 차별화는 장식 효과가 아니라 **제품 고유의 데이터 구조, 흐름, 증거 시각화**에서
  나온다.
- 사용자는 긴 설명을 읽기 전에 **TL;DR 형태의 전체 구조**를 먼저 보고 싶어 한다.
- 모바일 우선 랜딩페이지에서는 CTA, 폼, 핵심 혜택이 화면 상단과 하단에서
  반복 접근 가능해야 한다.

참고:

- Webflow, "8 web design trends to watch in 2026"
  https://webflow.com/blog/web-design-trends-2026
- Webstacks, "Minimalist Landing Page Design: 2026 Trends & Tips"
  https://www.webstacks.com/blog/minimalist-landing-page-design-trends
- Baymard Institute, mobile UX benchmark insights
  https://baymard.com/blog/mobile-ux-ecommerce

## 3. 핵심 디자인 원칙

### 원칙 1. 단순하지만 비어 보이면 안 된다

Apple식 미니멀은 정보가 없는 화면이 아니라, 중요한 정보만 남긴 화면이다.
이 제품은 건설 B2B SaaS이므로 추상적인 감성 문구보다 승인 흐름, 증빙 구조,
파일럿 범위가 명확해야 한다.

적용 기준:

- Hero에는 H1, 1문단 설명, 2개 CTA, MVP 범위 배지만 둔다.
- 첫 화면 아래쪽에 다음 섹션 일부가 보이게 하여 스크롤 가능성을 암시한다.
- 각 섹션은 한 가지 질문만 답한다.

### 원칙 2. 제품을 실제 업무 흐름처럼 보여준다

랜딩페이지의 메인 비주얼은 추상 일러스트가 아니라 "사진, 도면, 계근표,
계약 내역이 승인 데이터로 연결되는 화면"이어야 한다.

적용 기준:

- Hero 비주얼은 제품 대시보드/워크플로우 목업을 사용한다.
- 목업 안에는 실제 도메인 단어를 넣는다: `토공`, `내역번호`, `당월 수량`,
  `사진 증빙`, `감리 검수`.
- 화려한 3D 오브젝트, 의미 없는 그래디언트, 흐릿한 배경 이미지는 사용하지 않는다.

### 원칙 3. CTA는 조용하지만 분명해야 한다

CTA는 페이지의 유일한 주요 전환 행동이다.

적용 기준:

- Primary CTA: `우리 현장으로 파일럿 신청하기`
- Secondary CTA: `5분 데모 흐름 보기`
- Primary CTA는 첫 화면, 중간 ROI 이후, 마지막 폼 섹션에 반복한다.
- CTA 주변에는 경쟁 버튼을 두지 않는다.

### 원칙 4. 신뢰는 수치와 제한 조건에서 나온다

초기 제품은 모든 현장을 해결한다고 말하면 신뢰가 낮아진다. 오히려 제한 범위를
명확히 보여주는 편이 더 전문적으로 보인다.

적용 기준:

- `토공 1공종`, `LH 양식 우선`, `한 달 기성 사이클`, `사람 검수 후 확정`을
  첫 화면 또는 두 번째 화면에서 노출한다.
- `3일 -> 3시간 목표`, `감리 반려율 5% 미만 목표`, `월 9~19만 원 가설`은
  ROI 섹션에서 카드가 아닌 조밀한 지표 행으로 보여준다.

## 4. Visual System

### 4-1. 색상

전체 인상은 흰색 기반, 검정 텍스트, 절제된 블루 포인트다. 건설 도메인의
신뢰감을 위해 지나치게 파스텔톤이나 보라색 SaaS 느낌은 피한다.

| 용도             | 토큰                    | 값        |
| ---------------- | ----------------------- | --------- |
| 배경             | `--color-bg`            | `#F7F7F5` |
| 표면             | `--color-surface`       | `#FFFFFF` |
| 기본 텍스트      | `--color-text`          | `#111111` |
| 보조 텍스트      | `--color-muted`         | `#666A70` |
| 경계선           | `--color-border`        | `#E4E4E0` |
| 주요 CTA         | `--color-primary`       | `#0071E3` |
| CTA hover        | `--color-primary-hover` | `#005BB8` |
| 현장 데이터 강조 | `--color-data`          | `#1D7F5F` |
| 경고/반려        | `--color-risk`          | `#B54708` |

사용 규칙:

- 한 화면에서 강조색은 1개만 주로 사용한다.
- `--color-data`와 `--color-risk`는 제품 목업 안에서 상태 의미가 있을 때만 쓴다.
- 배경 그래디언트는 사용하지 않는다. 필요한 경우 아주 약한 단색 표면 차이로
  섹션을 구분한다.

### 4-2. 타이포그래피

시스템 폰트 기반으로 구현한다.

```css
font-family:
  -apple-system, BlinkMacSystemFont, "Segoe UI", "Noto Sans KR",
  "Apple SD Gothic Neo", sans-serif;
```

| 역할       | Desktop | Mobile | 굵기 | 줄간격 |
| ---------- | ------: | -----: | ---: | -----: |
| Hero H1    |    64px |   40px |  700 |   1.05 |
| Section H2 |    40px |   30px |  700 |   1.15 |
| Lead copy  |    22px |   18px |  400 |   1.55 |
| Body       |    17px |   16px |  400 |   1.65 |
| Caption    |    14px |   13px |  500 |   1.45 |
| Button     |    16px |   16px |  600 |      1 |

타이포그래피 규칙:

- 글자 간격은 `0`으로 둔다.
- Hero H1은 최대 2줄을 목표로 한다.
- 본문 문단은 2~3문장을 넘기지 않는다.
- 한국어 문장은 명사 나열보다 짧은 동사형 문장으로 쓴다.

### 4-3. 레이아웃

| 항목                | 기준                                 |
| ------------------- | ------------------------------------ |
| 최대 콘텐츠 폭      | 1120px                               |
| Hero 최대 폭        | 1200px                               |
| 본문 텍스트 최대 폭 | 680px                                |
| 섹션 상하 패딩      | Desktop 112px, Mobile 72px           |
| 그리드              | Desktop 12 columns, Mobile 4 columns |
| 카드 반경           | 8px 이하                             |
| 큰 제품 목업 반경   | 16px 이하                            |

레이아웃 규칙:

- Hero는 split card 레이아웃이 아니라, 메시지와 제품 시각화가 하나의 흐름처럼
  보이게 한다.
- 페이지 섹션을 카드 안에 넣지 않는다. 카드가 필요하면 반복 항목이나 제품
  목업 내부에만 사용한다.
- 모바일에서는 모든 2열 레이아웃을 1열로 전환한다.
- 버튼과 폼 입력은 최소 높이 44px 이상으로 둔다.

### 4-4. 모션

모션은 제품 이해를 돕는 곳에만 사용한다.

- 스크롤 진입 애니메이션은 opacity + y 12px 이하로 제한한다.
- 제품 목업에서 `사진 -> 내역번호 -> 승인자료` 연결이 순차적으로 강조되는
  정도는 허용한다.
- 무한 반복 배경 애니메이션, 과한 패럴랙스, 큰 블러 전환은 금지한다.
- `prefers-reduced-motion` 사용자는 애니메이션을 제거한다.

## 5. 페이지 구조

### 5-1. Hero

목적: 제품의 약속과 파일럿 행동을 즉시 전달한다.

필수 요소:

- H1: `현장 기록이 곧 기성 승인 자료가 되게 만듭니다`
- Subcopy: `사진, 도면, 계근표, 계약 내역을 승인 가능한 시공 데이터로 연결하고, 감리 제출용 기성 서류를 자동 생성합니다.`
- Primary CTA: `우리 현장으로 파일럿 신청하기`
- Secondary CTA: `5분 데모 흐름 보기`
- 범위 배지: `토공 1공종`, `LH 양식 우선`, `사람 검수 후 확정`
- 제품 목업: 승인 데이터 원장 또는 기성 패키지 생성 화면

금지:

- 추상적인 건설 현장 배경 사진 위에 텍스트만 올리는 방식
- "AI로 모든 기성 청구를 자동화"처럼 과장된 문구
- 첫 화면 안에 4개 이상의 CTA/링크

### 5-2. 문제

목적: 고객이 이미 겪는 월말 기성 업무의 마찰을 짧게 보여준다.

권장 구성:

- `사진`, `수량`, `도면`, `계근표`, `변경 증빙`이 분리되어 있는 상태를
  한 줄 흐름도로 표현한다.
- 각 페인은 한 문장으로만 쓴다.
- "서류 작성이 오래 걸림"보다 "감리/발주처가 다시 해석해야 함"을 강조한다.

### 5-3. 원인

목적: 단순 문서 자동화가 아니라 승인 데이터 구조화 문제임을 설명한다.

권장 구성:

- 계약 내역번호, 도면 위치, 당월 수량, 사진 증빙이 연결되지 않는 구조를
  before/after로 보여준다.

### 5-4. 솔루션

목적: 제품이 하는 일을 세 단계로 이해시킨다.

1. 현장 데이터 수집
2. 승인 데이터 구조화
3. 기성 서류 자동 생성

각 단계에는 한 줄 설명과 업무 산출물 예시만 둔다.

### 5-5. 파일럿 범위

목적: 제품의 현재 범위를 솔직하게 보여 신뢰를 만든다.

필수 항목:

- 토공 1공종
- 한 달 기성 청구 사이클
- LH 양식 우선
- AI 결과는 초안
- 공무 담당자 검수 후 확정
- 자동 제출 제외

### 5-6. ROI

목적: 결재권자에게 실험할 이유를 준다.

권장 지표:

- 기성 작성 시간: `3일 -> 3시간 목표`
- 감리 반려율: `< 5% 목표`
- 가격 가설: `현장당 월 9~19만 원`
- 가치: `작성 시간 절감 + 반려 리스크 감소 + 지급 지연 완화`

### 5-7. 파일럿 신청 폼

목적: 관심을 행동으로 전환한다.

폼 디자인 규칙:

- 필수 입력은 회사명, 역할, 이메일, 파일럿 관심 여부로 제한한다.
- 선택 질문은 3개 이하로 시작한다.
- 제출 실패 시 입력값을 유지한다.
- 제출 중 버튼 중복 클릭을 막는다.
- 성공 상태에서는 "연락 예정"과 "다음 단계"를 짧게 알려준다.

## 6. 컴포넌트 규칙

### Buttons

- Primary: 파란색 배경, 흰색 텍스트, 44px 이상 높이
- Secondary: 투명 배경, 텍스트 링크 또는 얇은 경계선
- 모바일에서는 Primary CTA를 너무 넓게 늘리지 말고 콘텐츠 폭 안에서 자연스럽게
  배치한다.

### Badges

- MVP 범위나 상태 표시용으로만 사용한다.
- 색상은 대부분 회색 계열을 사용하고, 중요한 상태만 데이터 강조색을 쓴다.

### Product Mockup

- 실제 업무 데이터처럼 보이는 텍스트를 넣는다.
- 목업은 장식 카드가 아니라 제품 이해를 돕는 설명 도구다.
- 모바일에서는 목업을 축소하기보다 핵심 행만 남겨 재구성한다.

### Forms

- 라벨은 항상 표시한다.
- placeholder에만 의미를 넣지 않는다.
- 오류 메시지는 필드 바로 아래에 둔다.
- 긴 선택지는 줄바꿈되어도 버튼 높이가 자연스럽게 늘어나야 한다.

## 7. 카피 톤

톤은 차분하고 구체적이어야 한다. "혁신", "완전 자동화", "AI가 다 해줍니다" 같은
표현은 피한다.

좋은 문장:

- `현장 사진과 당월 수량을 계약 내역번호에 연결합니다.`
- `AI가 초안을 만들고, 공무 담당자가 검수 후 확정합니다.`
- `감리 제출 전에 누락 증빙과 미확정 항목을 먼저 드러냅니다.`

피할 문장:

- `건설 산업을 완전히 바꿉니다.`
- `기성 청구를 100% 자동화합니다.`
- `클릭 한 번으로 모든 서류가 끝납니다.`

## 8. 접근성 및 성능 기준

- 텍스트 대비는 WCAG AA 수준을 목표로 한다.
- 모든 버튼과 링크는 키보드 포커스가 보여야 한다.
- CTA와 폼 입력은 터치 높이 44px 이상을 유지한다.
- 이미지에는 업무 의미가 있는 대체 텍스트를 제공한다.
- Hero의 핵심 메시지와 CTA는 모바일 375px 폭에서 겹치지 않아야 한다.
- 무거운 영상 자동재생은 사용하지 않는다.
- 첫 화면에는 외부 스크립트와 대용량 이미지를 최소화한다.
- 폼 제출은 일반 네트워크 기준 2초 내 완료를 목표로 한다.

## 9. 디자인 QA 체크리스트

구현 완료 전 다음을 확인한다.

- [ ] 첫 화면에서 `승인 가능 시공 데이터` 메시지가 즉시 보인다.
- [ ] Primary CTA가 첫 화면과 폼 섹션에서 명확히 보인다.
- [ ] Hero H1과 CTA가 375px 모바일 폭에서 겹치지 않는다.
- [ ] 제품 목업이 실제 업무 흐름을 설명한다.
- [ ] 범위 제한이 숨겨져 있지 않다.
- [ ] 폼 오류, 제출 중, 성공, 실패 상태가 모두 있다.
- [ ] 긴 회사명/역할/선택지 텍스트가 레이아웃을 깨지 않는다.
- [ ] 색상만으로 상태를 구분하지 않는다.
- [ ] `prefers-reduced-motion`에서 애니메이션이 꺼진다.
- [ ] 랜딩페이지가 보라색 SaaS 템플릿이나 추상 AI 툴처럼 보이지 않는다.

## 10. 구현 중 의사결정 기준

디자인 선택이 애매하면 다음 순서로 판단한다.

1. 파일럿 신청 전환에 도움이 되는가?
2. "승인 가능 시공 데이터" 포지셔닝을 더 명확히 만드는가?
3. 원도급/감리 사용자가 신뢰할 만큼 구체적인가?
4. 모바일에서 읽고 누르기 쉬운가?
5. 장식보다 제품 이해에 기여하는가?

위 질문 중 3개 이상에 답하지 못하는 요소는 제거한다.
